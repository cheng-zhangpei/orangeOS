## ch2-1 特权级机制

我们后续想要转变一下对一个这种基础软件学习的思路，将其转变为一种类似纯数学的思维，就是我们知道现在os有内核态和用户态，但是为啥要有这种分级，这个分级在cs中一开始为啥有这个东西？

特权级的来源其实就是os安全，就是os无法确保所有的程序都值得性能，所以一个很明显的想法就是将应用程序和内核程序隔离开来。

那么怎么做这个隔离？一个解决方法就是让os运行在一个安全的硬件的环境，所以这也是为啥cpu里面会有特权位的原因是吧。

- 应用程序不能访问任意的地址空间（这个在第四章会进一步讲解，本章不会涉及）
- 应用程序不能执行某些可能破坏计算机系统的指令（本章的重点）

所以，这种思维是不是很虚拟机级别的隔离非常像呢？有点意思啊，所以os安全很大程序上是os需要对外来的内容进行隔离。

> 其实我突然意识到一件事情，纯数的思维的确对一个人视角的建立会有完全不同的帮助。在学东西的时候多问为啥，绝对是有巨大的好处的。

为了实现这样的特权级机制，需要进行软硬件协同设计。一个比较简洁的方法就是，处理器设置两个不同安全等级的执行环境：用户态特权级的执行环境和内核态特权级的执行环境。且明确指出可能破坏计算机系统的内核态特权级指令子集，规定内核态特权级指令子集中的指令只能在内核态特权级的执行环境中执行。处理器在执行指令前会进行特权级安全检查，如果在用户态执行环境中执行这些内核态特权级指令，会产生异常。

我是否可以这样理解，直接在硬件上隔离的确方便，但是我们在进行隔离改动的时候，我们要考虑到现代通用的云架构，也就是不能改动太大导致产商无法轻松的部署该应用。

> rust的所有权机制：
>
> - 所有权（Ownership）
>
> **每个值在同一时刻只有一个变量拥有它**。
>
> 当变量离开作用域，它拥有的值会被自动销毁（释放内存）。
>
> 所有权可以转移（比如函数传参、赋值），转移后原变量不再可用。
>
> ```rust
> let s1 = String::from("hello");
> let s2 = s1;          // 所有权从 s1 转移到 s2
> println!("{}", s1);   // 编译错误！s1 已经失效
> ```
>
> - 借用（Borrowing）—— 不转移所有权，只借来用
>
> 用 `&` 或 `&mut` 创建**引用**，可以临时访问值而不取得所有权。
>
> `&T`：不可变引用（只读），可以有多个同时存在。
>
> `&mut T`：可变引用（读写），同一时间只能有一个。
>
> ```rust
> let mut x = 5;
> let r1 = &x;      // 不可变借用
> let r2 = &x;      // 可以多个
> // let r3 = &mut x; // 错误！不能同时有不可变和可变借用
> ```
>
> - 运行时借用检查（`RefCell`）
>
> 编译时检查太严格，有时需要在运行时动态检查。`RefCell<T>` 允许你在运行时进行借用检查：
>
> `.borrow()` 获得不可变引用（多个同时）
>
> `.borrow_mut()` 获得可变引用（只能一个）

---



> 所有权模型的设计目的：
>
> - **每个值只有一个所有者** → 释放时不会重复释放。
> - **所有权转移（move）** → 避免悬垂指针（原变量失效）。
> - **借用规则**（读写互斥） → 避免数据竞争和迭代器失效。



### 内部可变性

在 rCore 中，有很多全局的数据结构（比如任务管理器、内存分配器等），需要在不同的函数中修改它们。Rust 的全局变量（`static`）默认是不可变的，你不能直接修改。通常的解决方法是：

- 用 `RefCell` 提供内部可变性。
- 但是 `RefCell` 不是 `Sync` 的，也就是说它不能安全地放在 `static` 变量中（因为 `static` 变量可能被多个线程访问，而 `RefCell` 没有做并发保护）。

rCore 目前是单核的（没有多线程并发），所以实际上不存在多个 CPU 同时访问的问题。因此，我们可以**手动告诉编译器**：这个 `RefCell` 可以安全地放在全局变量中，因为只有单核。这就是 `unsafe impl Sync for UPSafeCell<T> {}` 的作用。

### Trap机制

对于Trap我们是分为硬件视角和软件视角的。硬件做的事情会比较简单主要就是设计到CSR（状态控制寄存器）,软件部分就是操作系统做的事情了。

- 硬件

| CSR 名  | 该 CSR 与 Trap 相关的功能                                    |
| ------- | ------------------------------------------------------------ |
| sstatus | `SPP` 等字段给出 Trap 发生之前 CPU 处在哪个特权级（S/U）等信息 |
| sepc    | 当 Trap 是一个异常的时候，记录 Trap 发生之前执行的最后一条指令的地址 |
| scause  | 描述 Trap 的原因                                             |
| stval   | 给出 Trap 附加信息                                           |
| stvec   | 控制 Trap 处理代码的入口地址                                 |

用户主动陷入的时候：

- `sstatus` 的 `SPP` 字段会被修改为 CPU 当前的特权级（U/S）。
- `sepc` 会被修改为被中断或触发异常的指令的地址。如 CPU 执行 `ecall` 指令会触发异常，则 `sepc` 会被设置为 `ecall` 指令的地址。
- `scause/stval` 分别会被修改成这次 Trap 的原因以及相关的附加信息。
- CPU 会跳转到 `stvec` 所设置的 Trap 处理入口地址，并将当前特权级设置为 S ，然后从Trap 处理入口地址处开始执行。

陷入返回的时候：

- CPU 会将当前的特权级按照 `sstatus` 的 `SPP` 字段设置为 U 或者 S ；
- CPU 会跳转到 `sepc` 寄存器指向的那条指令，然后继续执行。

> 其实可以很明显发现，我们只是让cpu所执行的位置回到了原来触发中断的地方，但是这个时候通用寄存器的实际的状态其实是丢失了，也就是之前执行的状态没掉了，所以这就是为啥我们要引入内核栈和用户栈的原因
>
> 那么现在问题是我使用一个栈不够吗？为啥呢，因为我需要进行隔离，我不能让用户拿到内核中的信息，如果共享栈，内核信息容易被攻击者利用。
>
> 现在我们的os还没有一个虚拟内存系统，所以不管是内核栈和用户栈都是共享同一套的虚拟内存的



> Trap 处理的总体流程如下：首先通过 `__alltraps` 将 Trap 上下文保存在内核栈上，然后跳转到使用 Rust 编写的 `trap_handler` 函数完成 Trap 分发及处理。当 `trap_handler` 返回之后，使用 `__restore` 从保存在内核栈上的 Trap 上下文恢复寄存器。最后通过一条 `sret` 指令回到应用程序执行。
